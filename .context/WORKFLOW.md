# 🔄 智能体协作工作流程 v2.0（所有智能体必读）

> 本文档定义了所有智能体在任务协作中的职责和流程。每个智能体必须遵守。
> 
> **v2.0 更新**：引入阶段流水线、依赖关系、评审门禁

---

## 一、核心改进

### v1.0 问题
- ❌ 任务分解后直接并行执行
- ❌ 开发在设计前就开始
- ❌ 测试在开发前就开始
- ❌ 缺少评审环节

### v2.0 解决
- ✅ 引入阶段流水线（需求→设计→开发→测试→部署）
- ✅ 引入依赖关系（后阶段依赖前阶段）
- ✅ 引入评审门禁（阶段完成后需评审确认）

## 二、项目上下文（重要）

**每个项目都有 `.context/` 目录，包含所有智能体必须了解的公共信息。**

### 上下文文件

| 文件 | 说明 | 创建者 |
|------|------|--------|
| `.context/WORKFLOW.md` | 完整工作流程 | 头号小弟（复制） |
| `.context/CONVENTIONS.md` | 关键约定 | 头号小弟（复制） |
| `.context/current-stage.json` | 当前阶段信息 | orchestrator 更新 |
| `.context/README.md` | 使用说明 | 头号小弟（创建） |

### 创建时机

头号小弟在创建项目目录时，调用脚本初始化：
```bash
~/.openclaw/workspace/scripts/init-project-context.sh <项目名> <任务描述>
```

### 使用方式

**子智能体启动时必须读取**：
1. `.context/WORKFLOW.md` - 了解工作流程
2. `.context/current-stage.json` - 了解当前阶段
3. `.context/CONVENTIONS.md` - 了解必须遵守的约定

**子智能体启动时必须检查前置文档**：

```python
# 读取当前阶段信息
stage_info = read_json(".context/current-stage.json")
current_stage = stage_info["current_stage"]
prerequisites = stage_info["stages"][current_stage]["prerequisites"]

# 检查前置文档是否存在
for prereq in prerequisites:
    if not file_exists(prereq["file"]):
        # 前置文档不存在，拒绝执行，汇报问题
        report("前置文档不存在: " + prereq["file"])
        exit("前置条件不满足")
```

### 前置文档对照表

| 阶段 | 前置文档 | 检查类型 |
|------|----------|----------|
| 需求分析 | 无 | - |
| 系统设计 | `doc/requirements.md` | exists_and_reviewed |
| 开发实现 | `doc/architecture.md` | exists_and_reviewed |
|  | `doc/api-spec.md` | exists |
|  | `doc/database-schema.md` | exists |
| 系统测试 | `code/` | exists |
|  | `doc/api-spec.md` | exists |
| 部署上线 | `doc/test-report.md` | exists_and_reviewed |

**检查类型说明**：
- `exists`：文档存在即可
- `exists_and_reviewed`：文档存在且前置阶段已评审通过

## 四、评审门禁机制

### 评审时机

| 评审点 | 触发条件 | 评审内容 |
|--------|----------|----------|
| 需求评审 | 阶段1完成 | 需求是否完整、清晰、可实现 |
| 设计评审 | 阶段2完成 | 架构是否合理、技术栈是否恰当 |
| 代码评审 | 阶段3每个功能点 | 代码质量、是否按设计实现 |
| 测试评审 | 阶段4完成 | 测试覆盖率、是否可以上线 |
| 上线评审 | 阶段5完成 | 系统是否正常运行 |

### 评审流程

```
阶段完成
    │
    ▼
orchestrator 汇报
    │
    ▼
头号小弟 转发汇报
    │
    ▼
老大评审决策
    │
    ├── ✅ 通过 → 进入下一阶段
    │
    ├── ❌ 不通过 + 修改意见 → 返回当前阶段
    │
    └── 💬 需要讨论 → 安排评审会议
```

### 评审结果处理

```python
# 评审通过
PUT /api/v1/tasks/{task_id}
{
    "status": "running",
    "result": {
        "current_stage": "系统设计",
        "stage_1_status": "passed",
        "stage_1_reviewed_at": "2026-03-23T12:00:00+08:00"
    }
}

# 评审不通过
PUT /api/v1/tasks/{task_id}
{
    "status": "blocked",
    "result": {
        "current_stage": "需求分析",
        "review_feedback": "需求缺少XXX，需要补充"
    }
}
```

## 五、依赖检查机制

### orchestrator 在派发任务前检查

```python
def can_start_stage(stage, task_state):
    """检查阶段是否可以启动"""
    
    dependencies = {
        "需求分析": [],
        "系统设计": ["需求分析"],
        "开发实现": ["系统设计"],
        "系统测试": ["开发实现"],
        "部署上线": ["系统测试"]
    }
    
    for dep in dependencies[stage]:
        if task_state[f"stage_{dep}_status"] != "passed":
            return False, f"依赖阶段 {dep} 未完成评审"
    
    return True, "可以启动"
```

### 依赖检查流程

```
orchestrator 收到任务
    │
    ▼
检查当前阶段依赖
    │
    ├── 依赖未完成 → 等待前置阶段完成
    │
    └── 依赖已完成 → 派发智能体执行
```

## 七、特殊情况处理

### 1. 小改动任务（跳过某些阶段）

```python
# orchestrator 分析任务类型
if 任务类型 == "简单bug修复":
    stages = ["开发实现", "系统测试"]  # 跳过需求、设计
elif 任务类型 == "文档更新":
    stages = ["文档更新"]  # 只需要文档
elif 任务类型 == "新功能开发":
    stages = ["需求分析", "系统设计", "开发实现", "系统测试", "部署上线"]
```

### 2. 紧急修复（快速通道）

```python
if 任务类型 == "紧急修复":
    # 跳过评审，直接执行
    stages = ["开发实现", "部署上线"]
    review_required = False  # 无需评审
```

### 3. 测试发现问题

```
测试智能体发现问题
    │
    ▼
提交 GitHub Issue
    │
    ▼
汇报给 orchestrator
    │
    ▼
orchestrator 协调开发智能体修复
    │
    ▼
开发智能体修复并提交
    │
    ▼
返回测试阶段重新测试
```

## 九、关键约定（更新）

| 约定 | 说明 |
|------|------|
| **阶段顺序** | 必须按阶段流水线顺序执行，不可跳过前置阶段 |
| **依赖检查** | 派发任务前必须检查依赖阶段是否完成 |
| **评审门禁** | 关键阶段完成后必须评审才能进入下一阶段 |
| **Git 提交** | 开发智能体必须边开发边提交 |
| **向上反馈** | 每个阶段完成或遇到问题必须立即汇报 |
| **监控入库** | 所有阶段状态变化必须写入数据库 |
| **技术栈** | 严格按照设计文档实现 |
